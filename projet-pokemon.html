Projet Académique

Pokémon Trading Card Game

Développement d'un moteur de jeu et d'une interface graphique JavaFX

17.86/20 Note Finale
3 Semaines Durée (IHM)
Binôme Équipe
Java
JavaFX
POO
TDD

01. Contexte

Type de projet : Projet Académique (BUT Informatique - SAE 2.01 & S2.01)

Durée : 3 semaines pour la phase graphique (Mai-Juin 2025)

Mode de travail : Binôme (Équipe de 2 étudiants)

Résultat : Note de 17.86/20 (132/154 tests unitaires validés sur la phase 1)

Objectifs & Contraintes

  • Implémenter les règles métiers complexes du jeu de cartes Pokémon TCG en Java pur (Architecture Orientée Objet stricte).
  • Créer une interface graphique interactive avec JavaFX respectant le pattern MVC.
  • Utilisation de Tests Unitaires (JUnit) pour valider la logique avant l'interface.
  • Interdiction de modifier les signatures imposées par le squelette fourni (Façades).

02. Contribution Personnelle

Phase 1 : Moteur de Jeu (Backend)

Java / TDD

J'ai été responsable de l'orchestration globale de la partie et des mécaniques "spéciales" des cartes :

  • Moteur de Jeu (`Jeu`) : Développement de la classe centrale gérant le flux de la partie (boucle de jeu), l'alternance des tours, l'arbitrage des règles et la détection des conditions de victoire/défaite.
  • Système de Talents (`Talents`) : Implémentation des capacités spéciales des Pokémon (effets passifs ou actifs activables par le joueur), nécessitant une gestion fine des états du jeu.
  • Cartes Dresseurs (`Dresseurs`) : Codage de la logique des cartes "Support" et "Objet" (ex: Potion, Changement, Pioche), impliquant des interactions directes avec le deck, la main et le banc.
  • Co-développement (`Joueur` & Tests) :
    • Collaboration sur la classe `Joueur` pour gérer la main et les actions de base.
    • Réalisation conjointe des tests unitaires (JUnit) pour assurer la fiabilité du code (Debugging croisé en fin de projet).

Phase 2 : Interface Graphique (Frontend)

JavaFX / MVC

Création de l'interface utilisateur et liaison avec le modèle via le pattern Façade.

  • Architecture IHM : Utilisation de conteneurs (`BorderPane`, `GridPane`) et création de composants personnalisés (`VueMain`, `VuePokemon`).
  • Data Binding : Utilisation de Listeners pour la mise à jour temps réel de l'interface (PV, cartes en main).
  • Événements : Gestion des clics (attaque, retraite, piocher) et flux d'instructions contextuel.

03. Apprentissages

Programmation Orientée Objet

Maîtrise avancée de l'héritage, du polymorphisme et de l'encapsulation pour gérer la complexité des cartes.

Architecture Logicielle

Compréhension du pattern Façade et MVC pour séparer proprement la logique métier de l'interface.

Qualité & Tests

Importance du TDD (Test Driven Development) avec 132 tests validés pour garantir la robustesse du code.

Travail d'Équipe

Gestion de version avec Git, résolution de conflits et collaboration efficace en binôme.
